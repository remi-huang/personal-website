写在前面：为什么写这篇 Vibe Coding 教程

这是我用 AI 搭建的第一个 iOS App，也是我完整跑通“从 0 到 1 用 AI 做产品并上架 App Store”全流程的一次实战记录。 我想分享的不只是“我做了什么”，更是每一步具体怎么操作、用什么工具、踩了什么坑，尤其是 App Store 审核被拒后怎么处理。

对我来说，这个 App 真正的意义不在产品本身，而在于我跑通了 vibe coding → 设计 → 开发 → 提审 → 上架的完整闭环。整个过程我给自己定了一个硬约束：一周做完。

一周这个时间限制反过来约束了产品复杂度。如果做需要用户登录、数据库、服务器同步的产品，光这些“地基”就吃掉一周了。 所以做第一个 AI App，我强烈建议你也选简单 + 感兴趣的组合：一个本地运行、本地存储、不需要后端的小工具，是最适合用 vibe coding 跑通全流程的形态。

用 AI 做 iOS App 需要哪些工具

整个 vibe coding 流程，我主要用了这几样：

  • Antigravity：Google 出的 AI agent，是我最主要的 vibe coding 工具。和 Codex 这类工具相比，Google 模型在视觉设计感上更突出，做 UI 偏重的产品比较顺手。这是整个流程的核心。
  • Google Stitch：AI 原型设计工具，用来生成产品最初的界面原型。
  • Google AI Studio：把 Stitch 生成的内容导进去可以预览动态交互效果。
  • Xcode：苹果官方的 iOS 开发环境，任何想上架 App Store 的 iOS App 都绕不开它。
  • Apple 开发者账号：上架 App Store 的硬门槛，年费 99 美元。
  • 图片素材：作为视觉参考，帮助 AI 更准确地理解我要的唱片机风格。

用 AI 做 iOS App 的完整步骤

01

用 Google Stitch 生成产品原型

我先用 Google Stitch 把脑子里的产品形态做了出来。这一步是把“我想做一个什么样的 App” 从模糊的想法变成可以看的界面。对零基础用户来说，Stitch 的好处是完全不用画图、不用懂 Figma，自然语言描述就能出界面。

02

导出到 Google AI Studio 预览动态效果

Stitch 出来的是相对静态的原型，把它导出到 Google AI Studio 之后，可以预览到带交互的动态效果。 这一步的价值在于：你能直观判断这个 App 跑起来到底是不是你想要的感觉，比看静态图准很多。

03

打包项目下载到本地电脑

在 AI Studio 里调到比较满意的状态后，把整个项目打包下载到电脑。这一步是从“在线原型” 过渡到“本地开发”的关键一跳。

04

用 Antigravity 打开项目

把刚才下载的文件夹，用 Antigravity 打开。从这一步开始，AI agent 正式接管主要的开发工作， 正式进入 vibe coding 阶段。

05

让 AI 把项目变成可上架的 iOS App

这一步是整个 vibe coding 流程里最关键的一跳。我直接在 Antigravity 里问 AI： “怎么把这个产品变成一个可以上架到 App Store 的完整文件？”

然后跟着 AI 一步步做。它会让你下载 Xcode、配置 Apple 开发者账号、调整项目结构、生成 iOS 原生项目文件，你基本只要照着指示执行就行。

从这一步开始，整个开发就形成了双轨工作流：Antigravity 负责写代码、改逻辑；Xcode 负责实时预览、打包、签名、上传。 只有 Xcode 才能把项目构建成 iOS 可用的 IPA 格式，完成代码签名，然后通过 Xcode 工具链上传到 App Store Connect 走审核流程。

这就是为什么用 AI 做 iOS App 一定要装 Xcode：它是上架这件事的“最后一公里”，绕不过去。

06

用 AI 改样式：图片参考 + 精确数字

我希望整个 App 做成唱片机的视觉风格。这一步是 vibe coding 里最有意思的环节，方法是 “AI + 自然语言 + 图片参考”三件套：把我喜欢的唱片机图片直接发给 AI，告诉它“我想要的感觉是这样的”，然后用自然语言一点点调整。

改细节时我最常用的两类 prompt 指令是：位置类，比如“这个元素往左/右移动 20px”、“再往下挪 10px”； 大小类，比如“这个按钮放大 1.2 倍”、“字号缩小一点”。

vibe coding 改 UI 的核心心法：不要说“我觉得不太对”，AI 猜不出来。要给具体方向 + 具体数字，每一轮都能看见进展。

07

提交 App Store 审核，以及第一次被拒

代码和样式都搞定后，通过 Xcode 把构建产物上传到 App Store Connect，提交审核。第一次没过。

苹果给的拒绝理由：我在界面里放了一个“登录”按钮，但这个 App 实际上根本没有真正的登录功能。 它完全本地运行，所有数据都存在用户手机本地，换设备不会同步。我之所以放登录入口，本来是想着“以后可能加上”。

但 App Store 审核规则很明确：如果一个功能当下不可用，就不应该出现在 App 里。 这是非常常见的拒审原因，新手一定要避开。

修复方法很简单：把那个登录入口直接移除，重新提交，第二次就过了。

如果你也是第一次走 App Store 上架流程，强烈建议提前看一遍苹果官方的 App Store Review Guidelines，尤其是“功能完整性”和“占位功能”那几条。

用 AI 做产品的几点真实复盘

回头看，这一周我学到的远不止“做出了一个 App”。AI 时代，学习 AI 工具最快的方式就是从一个真实项目入手。 看再多 vibe coding 教程、收藏再多 prompt 模板，都不如自己真的做一个东西。对我来说，我希望我的项目是一个真正可用的产品， 而不只是停留在原型阶段。
